    Chapitre 2 : L'Atelier Event Storming - Guide Pratique

    Chapitre 2 : L’Atelier Event Storming - Guide Pratique link

    🎯 Objectif de ce Chapitre link

    Ce chapitre vous donne une méthodologie complète pour organiser et animer un atelier Event Storming efficace. Vous apprendrez :

    • Comment préparer l’atelier
    • Les 7 étapes détaillées
    • Les techniques d’animation
    • Comment exploiter les résultats

    🎪 Qu’est-ce qu’un Atelier Event Storming ? link

    L’Event Storming est un atelier de conception collaborative qui réunit :

    • Experts métier : Ils connaissent les processus
    • Développeurs : Ils comprennent les contraintes techniques
    • Product Owners : Ils définissent les priorités
    • Architectes : Ils conçoivent l’architecture

    Durée Recommandée link

    • Atelier complet : 4-6 heures
    • Atelier découverte : 2-3 heures
    • Atelier approfondissement : 1-2 heures par contexte

    📋 Préparation de l’Atelier link

    Matériel Nécessaire link

    1. Post-its de couleurs :

      • 🟠 Orange : Événements métier
      • 🔵 Bleu : Commandes/Actions
      • 🟡 Jaune : Acteurs/Personnes
      • 🟢 Vert : Systèmes externes
      • 🟣 Violet : Vues de lecture
      • ⚪ Blanc : Questions/Problèmes

    2. Support :

      • Mur blanc ou tableau blanc
      • Marqueurs de couleurs
      • Chronomètre
      • Appareil photo

    3. Préparation :

      • Inviter 6-12 personnes maximum
      • Préparer un espace de travail confortable
      • Avoir des exemples de post-its

    Invités Recommandés link

    • 1-2 Experts métier par contexte
    • 2-3 Développeurs seniors
    • 1 Product Owner
    • 1 Architecte (optionnel)
    • 1 Facilitateur (vous)

    🎯 Les 7 Étapes de l’Atelier link

    Étape 1 : Identifier les Événements (30-45 min) link

    Objectif : Découvrir ce qui se passe dans le domaine métier

    Instructions :

    1. Chaque participant écrit des événements sur des post-its orange
    2. Un événement = une phrase au passé : “PaymentProcessed”, “UserRegistered”
    3. Coller les post-its sur le mur
    4. Grouper les événements similaires
    5. Éliminer les doublons

    Exemples d’événements :

    - PaymentRequested
- PaymentProcessed
- PaymentFailed
- UserRegistered
- UserLoggedIn
- SubscriptionCreated
- SubscriptionExpired
- OrderPlaced
- OrderCancelled

    Questions à poser :

    • “Qu’est-ce qui se passe dans votre métier ?”
    • “Quels sont les moments importants ?”
    • “Que voulez-vous savoir qui s’est passé ?”

    Étape 2 : Identifier les Acteurs (15-20 min) link

    Objectif : Découvrir qui déclenche les événements

    Instructions :

    1. Pour chaque événement, identifier qui le déclenche
    2. Écrire les acteurs sur des post-its jaunes
    3. Placer les acteurs à côté des événements

    Types d’acteurs :

    • Personnes : Customer, Admin, Manager
    • Systèmes : PaymentGateway, EmailService
    • Processus : AutomatedProcess, CronJob

    Exemples d’acteurs :

    - Customer (personne)
- Admin (personne)
- PaymentGateway (système)
- EmailService (système)
- SubscriptionRenewalProcess (processus)

    Étape 3 : Identifier les Commandes (20-30 min) link

    Objectif : Découvrir les actions qui déclenchent les événements

    Instructions :

    1. Pour chaque événement, identifier l’action qui le déclenche
    2. Écrire les commandes sur des post-its bleus
    3. Placer les commandes entre les acteurs et les événements

    Format des commandes :

    • Verbe à l’impératif : “ProcessPayment”, “RegisterUser”
    • Une commande peut déclencher plusieurs événements
    • Un événement peut être déclenché par plusieurs commandes

    Exemples de commandes :

    - ProcessPayment
- RegisterUser
- LoginUser
- CreateSubscription
- CancelOrder
- RefundPayment

    Étape 4 : Identifier les Agrégats (30-45 min) link

    Objectif : Découvrir les entités métier principales

    Instructions :

    1. Grouper les événements par entité métier
    2. Identifier les agrégats (entités avec des règles métier)
    3. Dessiner des rectangles autour des groupes d’événements
    4. Nommer les agrégats

    Caractéristiques d’un agrégat :

    • Contient des règles métier
    • A un cycle de vie cohérent
    • Peut être modifié de manière atomique
    • A une identité unique

    Exemples d’agrégats :

    - Payment (événements liés aux paiements)
- User (événements liés aux utilisateurs)
- Subscription (événements liés aux abonnements)
- Order (événements liés aux commandes)

    Étape 5 : Identifier les Systèmes Externes (15-20 min) link

    Objectif : Découvrir les intégrations nécessaires

    Instructions :

    1. Identifier les systèmes externes nécessaires
    2. Écrire les systèmes sur des post-its verts
    3. Placer les systèmes à côté des commandes/événements

    Types de systèmes externes :

    • APIs : PaymentGateway, EmailService
    • Bases de données : UserDatabase, PaymentDatabase
    • Services : NotificationService, AuditService

    Exemples de systèmes externes :

    - Stripe (paiements)
- SendGrid (emails)
- Keycloak (authentification)
- PostgreSQL (base de données)
- ElasticSearch (recherche)

    Étape 6 : Identifier les Vues de Lecture (20-30 min) link

    Objectif : Découvrir les données nécessaires pour l’affichage

    Instructions :

    1. Identifier les vues de lecture nécessaires
    2. Écrire les vues sur des post-its violets
    3. Placer les vues à côté des événements qui les alimentent

    Types de vues :

    • Tableaux : UserList, PaymentList
    • Détails : UserProfile, PaymentDetails
    • Rapports : MonthlyReport, Analytics

    Exemples de vues :

    - UserDashboard
- PaymentHistory
- SubscriptionStatus
- OrderSummary
- AnalyticsReport

    Étape 7 : Concevoir l’Architecture (30-45 min) link

    Objectif : Organiser le code selon les contextes métier

    Instructions :

    1. Grouper les agrégats par contexte métier
    2. Identifier les Bounded Contexts
    3. Dessiner des rectangles autour des contextes
    4. Nommer les contextes

    Exemples de Bounded Contexts :

    - Authentication (User, Login, Registration)
- Accounting (Payment, Subscription, Invoice)
- Cloud (Instance, Service, Configuration)
- GenAI (Agent, Chat, Model, Tool)

    🎨 Techniques d’Animation link

    Gestion du Temps link

    • Timeboxing : Limiter chaque étape dans le temps
    • Pauses : 5-10 minutes entre les étapes
    • Récapitulatif : Faire le point à la fin de chaque étape

    Gestion des Participants link

    • Tour de table : Donner la parole à chacun
    • Questions ouvertes : “Qu’est-ce qui se passe quand… ?”
    • Validation : “Est-ce que tout le monde est d’accord ?”
    • Désaccord : Noter les points de désaccord sur des post-its blancs

    Gestion des Dérives link

    • Recentrage : “Revenons à l’étape en cours”
    • Questions : “Est-ce que c’est dans le scope ?”
    • Report : “On note ça pour plus tard”

    📊 Exploitation des Résultats link

    Immédiatement Après l’Atelier link

    1. Photographier le mur complet
    2. Transcrire les post-its dans un document
    3. Valider avec les participants
    4. Planifier les prochaines étapes

    Structuration du Code link

    src/
├── Authentication/          # Bounded Context
│   ├── Domain/
│   │   ├── User/           # Agrégat User
│   │   │   ├── User.php
│   │   │   ├── UserId.php
│   │   │   └── Events/
│   │   │       ├── UserRegistered.php
│   │   │       └── UserLoggedIn.php
│   │   └── Commands/
│   │       ├── RegisterUser.php
│   │       └── LoginUser.php
│   ├── Application/
│   └── Infrastructure/
├── Accounting/              # Bounded Context
│   ├── Domain/
│   │   ├── Payment/        # Agrégat Payment
│   │   └── Subscription/   # Agrégat Subscription
│   ├── Application/
│   └── Infrastructure/
