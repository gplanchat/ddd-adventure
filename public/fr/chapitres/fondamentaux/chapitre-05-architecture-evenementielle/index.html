    Chapitre 5 : Architecture Événementielle (Optionnel)

    Chapitre 5 : Architecture Événementielle (Optionnel) link

    🎯 Objectif de ce Chapitre link

    Ce chapitre vous introduit à l’architecture événementielle pour les systèmes avec intégrations multiples. Vous apprendrez :

    • Les concepts de l’Event-Driven Architecture
    • Les Domain Events et leur utilisation
    • Les patterns de collaboration
    • L’implémentation avec API Platform

    🌐 Qu’est-ce que l’Architecture Événementielle ? link

    Principe Fondamental link

    L’architecture événementielle découple les composants en utilisant des événements pour communiquer. Au lieu d’appels directs, les composants publient et s’abonnent à des événements.

    Avantages link

    ✅ Découplage : Les composants ne connaissent pas leurs dépendances ✅ Évolutivité : Facile d’ajouter de nouveaux composants ✅ Résilience : Un composant en panne n’affecte pas les autres ✅ Audit Trail : Tous les événements sont enregistrés ✅ Intégrations : Facile d’intégrer des systèmes externes

    Inconvénients link

    ❌ Complexité : Plus complexe que l’architecture synchrone ❌ Debugging : Plus difficile de tracer les flux ❌ Cohérence : Eventual consistency au lieu de cohérence forte ❌ Performance : Latence due à la communication asynchrone

    🎪 Domain Events link

    Qu’est-ce qu’un Domain Event ? link

    Un Domain Event représente quelque chose qui s’est passé dans le domaine métier. Il est immuable et contient toutes les informations nécessaires.

    // Exemple de Domain Event
class PaymentProcessed implements DomainEvent
{
    public function __construct(
        public readonly PaymentId $paymentId,
        public readonly Money $amount,
        public readonly DateTime $processedAt,
        public readonly CustomerId $customerId
    ) {}

    public function getEventId(): string
    {
        return $this->paymentId->toString();
    }

    public function getEventType(): string
    {
        return 'payment.processed';
    }

    public function getOccurredAt(): DateTime
    {
        return $this->processedAt;
    }
}

    Patterns de Domain Events link

    1. Événements Immutables link

    class UserRegistered implements DomainEvent
{
    public function __construct(
        public readonly UserId $userId,
        public readonly Email $email,
        public readonly UserName $name,
        public readonly DateTime $registeredAt
    ) {}

    // Pas de setters, seulement des getters
    public function getEventId(): string
    {
        return $this->userId->toString();
    }
}

    2. Événements avec Métadonnées link

    class OrderConfirmed implements DomainEvent
{
    public function __construct(
        public readonly OrderId $orderId,
        public readonly CustomerId $customerId,
        public readonly Money $total,
        public readonly DateTime $confirmedAt,
        public readonly array $metadata = []
    ) {}

    public function getMetadata(): array
    {
        return $this->metadata;
    }
}

    🚌 Event Bus link

    Qu’est-ce qu’un Event Bus ? link

    L’Event Bus est le mécanisme qui permet de publier et de s’abonner aux événements. Il découple les producteurs et consommateurs d’événements.

    Implémentation Simple link

    interface EventBus
{
    public function publish(DomainEvent $event): void;
    public function subscribe(string $eventType, callable $handler): void;
}}

class InMemoryEventBus implements EventBus
{
    private array $handlers = [];

    public function publish(DomainEvent $event): void
    {
        $eventType = $event->getEventType();
        
        if (isset($this->handlers[$eventType])) {
            foreach ($this->handlers[$eventType] as $handler) {
                $handler($event);
            }
        }
    }

    public function subscribe(string $eventType, callable $handler): void
    {
        $this->handlers[$eventType][] = $handler;
    }
}

    Implémentation avec Symfony Messenger link

    // Configuration Symfony
# config/packages/messenger.yaml
framework:
    messenger:
        buses:
            event_bus:
                default_middleware: allow_no_handlers
        routing:
            'App\Domain\Events\*': event_bus

// Event Handler
class PaymentProcessedHandler
{
    public function __construct(
        private EmailService $emailService,
        private AuditService $auditService
    ) {}

    public function __invoke(PaymentProcessed $event): void
    {
        // Envoyer un email de confirmation
        $this->emailService->sendPaymentConfirmation(
            $event->customerId,
            $event->amount
        );
        
        // Enregistrer l'audit
        $this->auditService->recordPayment($event);
    }
}

    🔄 Patterns de Collaboration link

    1. Event Sourcing link

    L’Event Sourcing stocke les événements comme source de vérité et reconstruit l’état à partir des événements.

    class Payment
{
    private array $events = [];

    public function process(Money $amount): void
    {
        $this->recordEvent(new PaymentProcessed(
            $this->id,
            $amount,
            new DateTime()
        ));
    }

    private function recordEvent(DomainEvent $event): void
    {
        $this->events[] = $event;
    }

    public function getEvents(): array
    {
        return $this->events;
    }

    public static function fromEvents(array $events): self
    {
        $payment = new self();
        
        foreach ($events as $event) {
            $payment->apply($event);
        }
        
        return $payment;
    }
}

    2. CQRS (Command Query Responsibility Segregation) link

    CQRS sépare les modèles de lecture et d’écriture.

    // Command Model (Écriture)
class PaymentCommand
{
    public function __construct(
        public readonly PaymentId $id,
        public readonly Money $amount,
        public readonly PaymentMethod $method
    ) {}
}}

// Query Model (Lecture)
class PaymentQuery
{
    public function __construct(
        public readonly PaymentId $id,
        public readonly Money $amount,
        public readonly PaymentStatus $status,
        public readonly DateTime $createdAt,
        public readonly ?DateTime $processedAt
    ) {}
}

    3. Saga Pattern link

    Le Saga Pattern gère les transactions distribuées en utilisant des événements.

    class OrderSaga
{
    public function __construct(
        private EventBus $eventBus
    ) {
        $this->eventBus->subscribe(OrderCreated::class, [$this, 'handleOrderCreated']);
        $this->eventBus->subscribe(PaymentProcessed::class, [$this, 'handlePaymentProcessed']);
        $this->eventBus->subscribe(InventoryReserved::class, [$this, 'handleInventoryReserved']);
    }

    public function handleOrderCreated(OrderCreated $event): void
    {
        // Démarrer le processus de paiement
        $this->eventBus->publish(new ProcessPaymentCommand(
            $event->orderId,
            $event->total
        ));
    }

    public function handlePaymentProcessed(PaymentProcessed $event): void
    {
        // Réserver l'inventaire
        $this->eventBus->publish(new ReserveInventoryCommand(
            $event->orderId,
            $event->items
        ));
    }
}

    🏗️ Intégration avec API Platform link

    Configuration des Événements link

    // Configuration API Platform
#[ApiResource]
class ProcessPaymentCommand
{
    public function __construct(
        public readonly PaymentId $paymentId,
        public readonly Money $amount,
        public readonly PaymentMethod $method
    ) {}
}}

// Command Handler
class ProcessPaymentHandler
{
    public function __construct(
        private PaymentRepository $repository,
        private EventBus $eventBus
    ) {}

    public function __invoke(ProcessPaymentCommand $command): void
    {
        $payment = $this->repository->find($command->paymentId);
        $payment->process($command->amount);
        
        $this->repository->save($payment);
        
        // Publier les événements
        foreach ($payment->getEvents() as $event) {
            $this->eventBus->publish($event);
        }
    }
}

    Projections de Lecture link

    class PaymentProjection
{
    public function __construct(
        private PaymentQueryRepository $repository
    ) {}

    public function handle(PaymentProcessed $event): void
    {
        $query = new PaymentQuery(
            $event->paymentId,
            $event->amount,
            PaymentStatus::PROCESSED,
            $event->processedAt,
            $event->processedAt
        );
        
        $this->repository->save($query);
    }
}

    🎯 Votre Prochaine Étape link

    Maintenant que vous comprenez l’architecture événementielle, quel est votre niveau de complexité et vos besoins ?

    🟢 Option A : Audit trail critique, debugging complexe, mais modèles simples link

    Vous voulez Event Sourcing seul

    Critères :

    • Audit trail critique
    • Debugging complexe nécessaire
    • Modèles de lecture/écriture similaires
    • Équipe expérimentée avec Event Sourcing
    • Pas de besoin de modèles distincts

    Architecture recommandée : Event Sourcing seul Temps estimé : 2-3 mois d’apprentissage

    → Aller au Chapitre 7 (Event Sourcing - La Source de Vérité)

    🟡 Option B : Lectures/écritures différentes, mais un seul modèle suffit link

    Vous voulez CQS (Command Query Separation)

    Critères :

    • Lectures/écritures différentes mais modèles similaires
    • Besoin de performance sans complexité CQRS
    • Équipe intermédiaire
    • Un seul modèle riche suffit
    • Intégration possible avec Event Sourcing

    Architecture recommandée : CQS Temps estimé : 2-3 semaines d’apprentissage

    → Aller au Chapitre 8 (Architecture CQS - Command Query Separation)

    🔴 Option C : Lectures/écritures très différentes, modèles distincts nécessaires link

    Vous voulez CQRS

    Critères :

    • Lectures/écritures très différentes
    • Modèles de lecture/écriture distincts nécessaires
    • Équipes séparées (lecture/écriture)
    • Performance critique
    • Équipe expérimentée

    Architecture recommandée : CQRS Temps estimé : 1-2 mois d’apprentissage

    → Aller au Chapitre 9 (Architecture CQRS avec API Platform)

    ⚡ Option D : Système très complexe, audit trail + modèles distincts link

    Vous voulez CQRS + Event Sourcing Combinés

    Critères :

    • Audit trail critique
    • Performance critique sur les lectures
    • Modèles de lecture/écriture très différents
    • Évolution fréquente des vues métier
    • Équipe très expérimentée
    • Budget et temps importants

    Architecture recommandée : Event Sourcing + CQRS Temps estimé : 4-6 mois d’apprentissage

    → Aller au Chapitre 10 (CQRS + Event Sourcing Combinés)

    ⚠️ Option E : Je veux comprendre la granularité des choix architecturaux link

    Vous voulez savoir à quel niveau faire vos choix

    Critères :

    • Équipe avec plusieurs contextes
    • Besoin de flexibilité
    • Choix architecturaux complexes
    • Équipe expérimentée

    Temps estimé : 15-25 minutes

    → Aller au Chapitre 3.1 (Granularité des Choix Architecturaux)

    🔄 Option F : Je veux d’abord voir les options de stockage link

    Vous voulez comprendre les patterns de repository

    Critères :

    • Besoin de comprendre la persistance
    • Implémentation technique importante
    • Choix de stockage à faire
    • Patterns de repository à maîtriser

    Temps estimé : 30-45 minutes

    → Aller au Chapitre 6 (Repositories et Persistance)

    💡 Conseil : Si vous n’êtes pas sûr, choisissez l’option B (CQS) pour commencer avec une approche intermédiaire, puis évoluez vers des patterns plus complexes si nécessaire.
