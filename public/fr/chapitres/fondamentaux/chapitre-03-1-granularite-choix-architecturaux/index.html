    Chapitre 3.1 : Granularité des Choix Architecturaux

    Chapitre 3.1 : Granularité des Choix Architecturaux link

    🎯 Objectif de ce Chapitre link

    Ce chapitre vous aide à comprendre à quel niveau faire vos choix architecturaux. Vous apprendrez :

    • Les trois niveaux de granularité architecturale
    • Les critères pour chaque niveau
    • Les avantages et inconvénients de chaque approche
    • Comment gérer la charge mentale

    🏗️ Les Trois Niveaux de Granularité link

    1. Choix Global (Application) link

    Définition : Une architecture unique pour toute l’application

    Caractéristiques :

    • Architecture cohérente partout
    • Charge mentale minimale
    • Formation simplifiée
    • Maintenance facilitée

    Exemple : “Notre application utilise CQRS globalement”

    2. Choix par Bounded Context (Domaine) link

    Définition : Architecture spécifique par domaine métier

    Caractéristiques :

    • Optimisation par domaine
    • Équipes autonomes
    • Évolution indépendante
    • Performance adaptée

    Exemple : “Le contexte Accounting utilise Event Sourcing, Authentication utilise CQS”

    3. Choix par Agrégat (Entité) link

    Définition : Architecture fine pour des entités spécifiques

    Caractéristiques :

    • Optimisation maximale
    • Flexibilité totale
    • Performance adaptée
    • Audit trail ciblé

    Exemple : “L’agrégat Payment utilise Event Sourcing, User utilise Classique”

    📊 Matrice de Décision par Granularité link

    Critère Global Bounded Context Agrégat
    Équipe 2-8 devs 5-15 devs 8+ devs
    Cohérence Maximale Modérée Faible
    Flexibilité Faible Moyenne Élevée
    Charge Mentale Faible Moyenne Élevée
    Temps de Décision 1-2 semaines 2-4 semaines 4-8 semaines
    Maintenance Simple Modérée Complexe

    🌐 Choix Global (Application) link

    Quand Choisir le Niveau Global link

    Critères :

    • Équipe petite à moyenne (2-8 développeurs)
    • Application cohérente et homogène
    • Budget et temps limités
    • Maintenance simplifiée
    • Formation de l’équipe nécessaire

    Avantages link

    ✅ Cohérence maximale : Même architecture partout ✅ Charge mentale minimale : Un seul pattern à maîtriser ✅ Formation simplifiée : Formation unique pour toute l’équipe ✅ Maintenance facilitée : Même approche partout ✅ Développement rapide : Pas de décisions architecturales

    Inconvénients link

    ❌ Performance non optimale : Architecture non adaptée partout ❌ Flexibilité limitée : Difficile d’évoluer ❌ Évolution coûteuse : Changer d’architecture = refactoring global ❌ Limitation technique : Certains domaines peuvent nécessiter d’autres approches

    Exemple d’Architecture Globale link

    // Toute l'application utilise CQRS
src/
├── Authentication/
│   ├── Command/          # CQRS partout
│   └── Query/
├── Accounting/
│   ├── Command/          # CQRS partout
│   └── Query/
└── Cloud/
    ├── Command/          # CQRS partout
    └── Query/

    🏢 Choix par Bounded Context (Domaine) link

    Quand Choisir le Niveau Bounded Context link

    Critères :

    • Équipe moyenne à grande (5-15 développeurs)
    • Domaines métier distincts
    • Besoins différents par contexte
    • Équipes dédiées par domaine
    • Évolutivité importante

    Avantages link

    ✅ Optimisation par domaine : Architecture adaptée aux besoins ✅ Équipes autonomes : Chaque équipe maîtrise son domaine ✅ Évolution indépendante : Chaque contexte évolue séparément ✅ Performance adaptée : Architecture optimisée par usage ✅ Flexibilité modérée : Possibilité d’évoluer par contexte

    Inconvénients link

    ❌ Charge mentale modérée : Plusieurs patterns à maîtriser ❌ Interfaces à définir : Communication entre contextes ❌ Coordination nécessaire : Gestion des intégrations ❌ Formation multiple : Formation par contexte

    Exemple d’Architecture par Bounded Context link

    // Chaque contexte a sa propre architecture
src/
├── Authentication/          # Contexte simple
│   ├── Domain/              # Architecture classique
│   └── Infrastructure/
├── Accounting/              # Contexte complexe
│   ├── Command/             # CQRS
│   ├── Query/
│   └── Events/              # Event Sourcing
└── Cloud/                   # Contexte intermédiaire
    ├── Domain/              # CQS
    └── Infrastructure/

    🔧 Choix par Agrégat (Entité) link

    Quand Choisir le Niveau Agrégat link

    Critères :

    • Équipe expérimentée (8+ développeurs)
    • Entités avec besoins très spécifiques
    • Performance critique sur certaines entités
    • Audit trail nécessaire sur certaines entités
    • Budget et temps importants

    Avantages link

    ✅ Optimisation maximale : Architecture parfaitement adaptée ✅ Flexibilité totale : Chaque entité peut avoir sa propre architecture ✅ Performance adaptée : Optimisation fine par entité ✅ Audit trail ciblé : Event Sourcing seulement où nécessaire

    Inconvénients link

    ❌ Charge mentale élevée : Nombreux patterns à maîtriser ❌ Complexité de maintenance : Plusieurs architectures à maintenir ❌ Formation approfondie : Formation sur tous les patterns ❌ Coordination complexe : Gestion des interactions entre patterns

    Exemple d’Architecture par Agrégat link

    // Chaque agrégat a sa propre architecture
src/
├── Authentication/
│   ├── User/                # Agrégat simple - Classique
│   │   ├── User.php
│   │   └── UserRepository.php
│   └── Session/             # Agrégat complexe - Event Sourcing
│       ├── Session.php
│       ├── Events/
│       └── Projections/
├── Accounting/
│   ├── Payment/             # Agrégat critique - Event Sourcing + CQRS
│   │   ├── Command/
│   │   ├── Query/
│   │   └── Events/
│   └── Invoice/             # Agrégat simple - CQS
│       ├── Invoice.php
│       └── InvoiceRepository.php

    🧠 Gestion de la Charge Mentale link

    Principe de Cohérence Architecturale link

    • Maximum 3 architectures différentes dans le même système
    • Préférer la cohérence à la performance optimale
    • Documenter clairement les choix par Bounded Context
    • Éviter les transitions architecturales fréquentes

    Règles de Cohabitation link

    ✅ Combinaisons Recommandées link

    • Classique + CQS : Transition douce, charge mentale acceptable
    • CQS + Event Sourcing : Bon compromis complexité/performance
    • CQRS + Event Sourcing : Pour les équipes très expérimentées

    ⚠️ Combinaisons à Éviter link

    • Plus de 2 architectures dans le même Bounded Context
    • Event Sourcing + CQRS + Temporal : Charge mentale excessive
    • Mélange de patterns sans documentation claire

    ❌ Combinaisons Interdites link

    • Plus de 3 architectures dans le même système
    • Patterns contradictoires (ex: Classique + Event Sourcing dans le même agrégat)
    • Architectures non documentées ou mal justifiées

    Stratégies de Réduction de Charge Mentale link

    1. Simplification Progressive link

    • Commencer par l’architecture la plus simple
    • Ajouter la complexité progressivement
    • Mesurer l’impact à chaque étape

    2. Formation et Documentation link

    • Formation approfondie de l’équipe
    • Documentation claire et à jour
    • Pair programming sur les patterns complexes

    3. Outillage et Automatisation link

    • Outils de développement adaptés
    • Automatisation des tâches répétitives
    • Monitoring et observabilité

    4. Découpage et Isolation link

    • Isoler les parties complexes
    • Interfaces claires entre les architectures
    • Tests d’intégration robustes

    🎯 Votre Prochaine Étape link

    Maintenant que vous comprenez la granularité des choix architecturaux, à quel niveau voulez-vous faire vos choix ?

    🌐 Option A : Choix Global (Application) link

    Architecture unique pour toute l’application

    Critères :

    • Équipe petite à moyenne (2-8 développeurs)
    • Application cohérente et homogène
    • Budget et temps limités
    • Maintenance simplifiée

    Avantages : Cohérence maximale, charge mentale minimale Inconvénients : Performance non optimale partout, flexibilité limitée

    Temps estimé : 1-2 semaines de décision

    → Retourner au Chapitre 3 (Complexité Accidentelle vs Essentielle)

    🏢 Option B : Choix par Bounded Context (Domaine) link

    Architecture spécifique par domaine métier

    Critères :

    • Équipe moyenne à grande (5-15 développeurs)
    • Domaines métier distincts
    • Besoins différents par contexte
    • Équipes dédiées par domaine

    Avantages : Optimisation par domaine, équipes autonomes Inconvénients : Charge mentale modérée, interfaces à définir

    Temps estimé : 2-4 semaines de décision

    → Aller au Chapitre 4 (Modèles Riches vs Anémiques) puis choisir par contexte

    🔧 Option C : Choix par Agrégat (Entité) link

    Architecture fine pour des entités spécifiques

    Critères :

    • Équipe expérimentée (8+ développeurs)
    • Entités avec besoins très spécifiques
    • Performance critique sur certaines entités
    • Audit trail nécessaire sur certaines entités

    Avantages : Optimisation maximale, flexibilité totale Inconvénients : Charge mentale élevée, complexité de maintenance

    Temps estimé : 4-8 semaines de décision

    → Aller au Chapitre 4 (Modèles Riches vs Anémiques) puis choisir par agrégat

    ⚠️ Option D : Je veux d’abord comprendre les implications link

    Comprendre les impacts avant de choisir

    Critères :

    • Équipe peu expérimentée
    • Projet complexe
    • Besoin de formation
    • Décision critique

    Temps estimé : 1-2 semaines de formation

    → Aller au Chapitre 6 (Repositories et Persistance) pour voir les implémentations

    💡 Conseil : Si vous n’êtes pas sûr, choisissez l’option A (Global) pour commencer simple, puis évoluez vers des choix plus granulaires si nécessaire.

    🔄 Alternative : Si vous voulez tout voir dans l’ordre, continuez avec le Chapitre 4.
