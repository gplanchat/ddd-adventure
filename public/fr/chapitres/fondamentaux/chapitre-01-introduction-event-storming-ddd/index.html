    🚨 Le Problème des Modèles Anémiques link

    Qu’est-ce qu’un Modèle Anémique ? link

    Un modèle anémique est un modèle de données qui ne contient que des propriétés (getters/setters) sans logique métier. Toute la logique est déplacée dans des services externes.

    // ❌ Modèle Anémique - Exemple typique
class User
{
    public function __construct(
        private string $id,
        private string $email,
        private string $firstName,
        private string $lastName,
        private bool $isActive = true
    ) {}

    // Seulement des getters/setters
    public function getId(): string { return $this->id; }
    public function getEmail(): string { return $this->email; }
    public function setEmail(string $email): void { $this->email = $email; }
    // ... autres getters/setters
}

    Pourquoi c’est Problématique ? link

    1. Perte de l’Intention Métier : Le code ne reflète plus les règles métier
    2. Violation du Principe DRY : La logique est dupliquée dans les services
    3. Difficile à Tester : Les règles métier sont éparpillées
    4. Évolution Complexe : Changer une règle métier nécessite de modifier plusieurs endroits

    Exemple Concret : Gestion des Paiements link

    // ❌ Approche Anémique
class PaymentService
{
    public function processPayment(Payment $payment, float $amount): void
    {
        // Logique métier dans le service
        if ($amount <= 0) {
            throw new InvalidArgumentException('Amount must be positive');
        }
        
        if ($payment->getStatus() !== 'pending') {
            throw new InvalidStateException('Payment already processed');
        }
        
        if ($amount > $payment->getMaxAmount()) {
            throw new BusinessRuleException('Amount exceeds maximum');
        }
        
        // Plus de logique métier...
        $payment->setStatus('processed');
        $payment->setProcessedAt(new DateTime());
        $payment->setAmount($amount);
    }
}

    🎯 La Solution : Domain-Driven Design link

    Principe Fondamental link

    Le DDD place la logique métier au cœur du modèle, dans les entités et objets de valeur.

    // ✅ Modèle Riche - Approche DDD
class Payment
{
    private function __construct(
        private PaymentId $id,
        private Money $amount,
        private PaymentStatus $status,
        private ?DateTime $processedAt = null
    ) {}

    public static function create(PaymentId $id, Money $amount): self
    {
        if ($amount->isNegative()) {
            throw new InvalidAmountException('Payment amount cannot be negative');
        }
        
        return new self($id, $amount, PaymentStatus::PENDING);
    }

    public function process(Money $amount): void
    {
        if (!$this->canBeProcessed()) {
            throw new PaymentCannotBeProcessedException();
        }
        
        if ($amount->isGreaterThan($this->getMaxAllowedAmount())) {
            throw new AmountExceedsMaximumException();
        }
        
        $this->amount = $amount;
        $this->status = PaymentStatus::PROCESSED;
        $this->processedAt = new DateTime();
    }

    private function canBeProcessed(): bool
    {
        return $this->status === PaymentStatus::PENDING;
    }

    private function getMaxAllowedAmount(): Money
    {
        // Logique métier pour calculer le montant maximum
        return $this->amount->multiply(1.1); // 10% de marge
    }
}

    Avantages du DDD link

    1. Intention Métier Préservée : Le code reflète les règles métier
    2. Cohérence : Les règles sont centralisées dans le modèle
    3. Testabilité : Facile de tester les règles métier
    4. Évolutivité : Changer une règle ne nécessite qu’une modification

    🎪 L’Event Storming : Une Méthode de Conception Collaborative link

    Qu’est-ce que l’Event Storming ? link

    L’Event Storming est une méthode de conception collaborative qui permet de :

    • Découvrir le domaine métier
    • Identifier les événements métier
    • Modéliser les processus
    • Concevoir l’architecture

    Les 7 Étapes de l’Event Storming link

    1. Identifier les Événements : Qu’est-ce qui se passe dans le domaine ?
    2. Identifier les Acteurs : Qui déclenche ces événements ?
    3. Identifier les Commandes : Quelles actions déclenchent les événements ?
    4. Identifier les Agrégats : Quelles entités sont concernées ?
    5. Identifier les Systèmes Externes : Quelles intégrations sont nécessaires ?
    6. Identifier les Vues de Lecture : Quelles données sont nécessaires pour l’affichage ?
    7. Concevoir l’Architecture : Comment organiser le code ?

    Exemple d’Event Storming : Système de Paiement link

    Événements (Post-its Orange) :
- PaymentRequested
- PaymentProcessed
- PaymentFailed
- PaymentRefunded

Acteurs (Post-its Jaunes) :
- Customer
- PaymentGateway
- Admin

Commandes (Post-its Bleus) :
- ProcessPayment
- RefundPayment
- CancelPayment

Agrégats (Post-its Jaunes avec bordure) :
- Payment
- Customer
- Order
