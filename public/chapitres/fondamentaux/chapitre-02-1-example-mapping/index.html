    Chapitre 2.1 : L'Example Mapping - Détailer les Règles Métier

    🎯 Objectif de ce Chapitre link

    Ce chapitre vous donne une méthodologie complète pour utiliser l’Example Mapping après l’Event Storming. Vous apprendrez :

    • Comment organiser un atelier d’Example Mapping
    • Les 4 types de cartes et leur utilisation
    • Comment transformer les exemples en tests d’acceptation
    • Les bonnes pratiques pour éviter l’over-engineering

    Qu’est-ce que l’Example Mapping ? link

    Définition link

    L’Example Mapping est une technique de conception collaborative créée par Matt Wynne qui permet d’explorer les règles métier complexes à travers des exemples concrets. Elle utilise des cartes colorées pour structurer la discussion et révéler les cas limites.

    Relation avec l’Event Storming link

    L’Example Mapping est le complément naturel de l’Event Storming :

    • Event Storming : Découvre les événements et les processus métier
    • Example Mapping : Détaille les règles qui gouvernent ces événements

    Quand Utiliser l’Example Mapping ? link

    ✅ Utilisez l’Example Mapping si : link

    • Vous avez identifié des événements complexes lors de l’Event Storming
    • Les règles métier ne sont pas claires
    • Il y a des cas limites ou des exceptions
    • L’équipe a des questions sur le comportement attendu

    ❌ Évitez l’Example Mapping si : link

    • Les règles métier sont triviales
    • L’équipe est déjà alignée sur le comportement
    • Vous n’avez pas d’expert métier disponible
    • Le temps est très limité

    Les 4 Types de Cartes link

    🟡 Cartes Jaunes : Règles Métier link

    Les règles qui gouvernent le comportement du système. Elles doivent être :

    • Spécifiques : Pas de règles vagues
    • Testables : On doit pouvoir les vérifier
    • Métier : Pas de règles techniques

    Exemples de Bonnes Règles link

    ✅ Un paiement ne peut être traité que s'il est en statut "pending"
✅ Le montant doit être positif
✅ Un utilisateur ne peut pas emprunter plus que sa limite de crédit
✅ Une commande expirée ne peut plus être modifiée

    Exemples de Mauvaises Règles link

    ❌ Le système doit être performant
❌ L'interface doit être intuitive
❌ Les données doivent être sécurisées
❌ Le code doit être maintenable

    🟢 Cartes Vertes : Exemples link

    Les exemples concrets qui illustrent les règles. Ils doivent être :

    • Concrets : Avec des valeurs réelles
    • Complets : Avec le contexte nécessaire
    • Testables : On doit pouvoir les automatiser

    Format d’un Bon Exemple link

    🟢 EXEMPLE : [Contexte] → [Action] → [Résultat attendu]

Exemples :
🟢 EXEMPLE : Paiement de 100€ avec compte ayant une limite de 500€
   → Résultat : Paiement traité avec succès

🟢 EXEMPLE : Paiement de -50€
   → Résultat : Erreur "Montant invalide"

🟢 EXEMPLE : Tentative de traiter un paiement déjà traité
   → Résultat : Erreur "Paiement déjà traité"

    🔴 Cartes Rouges : Questions link

    Les questions qui émergent lors de l’exploration. Elles révèlent :

    • Les zones d’incertitude
    • Les cas limites non couverts
    • Les dépendances externes
    • Les contraintes techniques

    Exemples de Questions Pertinentes link

    🔴 QUESTION : Que se passe-t-il si le compte est suspendu ?
🔴 QUESTION : Y a-t-il une limite de temps pour traiter un paiement ?
🔴 QUESTION : Peut-on traiter un paiement partiellement ?
🔴 QUESTION : Que se passe-t-il si la connexion réseau échoue ?

    🔵 Cartes Bleues : Scénarios link

    Les scénarios d’utilisation complets qui montrent le flux end-to-end.

    Format d’un Bon Scénario link

    🔵 SCÉNARIO : [Titre du scénario]
   1. [Étape 1]
   2. [Étape 2]
   3. [Étape 3]
   ...
   → [Résultat final]

Exemple :
🔵 SCÉNARIO : Processus complet de paiement
   1. Client initie un paiement de 100€
   2. Système valide le montant
   3. Système vérifie la limite du compte (500€)
   4. Système traite le paiement
   5. Système envoie une confirmation
   6. Client reçoit un email de confirmation
   → Résultat : Paiement traité avec succès

    Guide Pratique de l’Atelier link

    Préparation de l’Atelier link

    1. Choisir la Fonctionnalité link

    Sélectionnez une fonctionnalité découverte lors de l’Event Storming qui :

    • A des règles métier complexes
    • Génère des questions dans l’équipe
    • Est critique pour le business
    • A des cas limites identifiés

    2. Composer l’Équipe link

    • 1 Expert Métier : Connaît les règles business
    • 1-2 Développeurs : Comprendront l’implémentation
    • 1 Testeur : Pensera aux cas limites
    • 1 Facilitateur : Animer l’atelier

    3. Préparer le Matériel link

    • Cartes colorées (jaunes, vertes, rouges, bleues)
    • Post-its
    • Marqueurs
    • Table ou mur pour organiser les cartes

    Déroulement de l’Atelier link

    Phase 1 : Identifier les Règles (15-20 min) link

    1. L’expert métier explique la fonctionnalité
    2. L’équipe identifie les règles métier
    3. Chaque règle est écrite sur une carte jaune
    4. Le facilitateur organise les cartes sur la table

    Phase 2 : Créer les Exemples (20-25 min) link

    1. Pour chaque règle, l’équipe crée des exemples
    2. Chaque exemple est écrit sur une carte verte
    3. Les exemples sont placés sous leur règle
    4. L’équipe valide que les exemples couvrent tous les cas

    Phase 3 : Poser les Questions (10-15 min) link

    1. L’équipe identifie les zones d’incertitude
    2. Chaque question est écrite sur une carte rouge
    3. Les questions sont placées à côté des règles concernées
    4. L’expert métier répond aux questions possibles

    Phase 4 : Créer les Scénarios (10-15 min) link

    1. L’équipe identifie les scénarios principaux
    2. Chaque scénario est écrit sur une carte bleue
    3. Les scénarios sont placés en bas de la table
    4. L’équipe valide la cohérence des scénarios

    Exemple Complet : Système de Réservation link

    Règles Métier Identifiées link

    🟡 RÈGLE : Une réservation ne peut être créée que si la chambre est disponible
🟡 RÈGLE : Une réservation doit avoir une date d'arrivée et de départ
🟡 RÈGLE : La date de départ doit être après la date d'arrivée
🟡 RÈGLE : Une réservation ne peut pas dépasser 30 jours
🟡 RÈGLE : Une réservation annulée ne peut plus être modifiée
🟡 RÈGLE : Un client ne peut avoir qu'une seule réservation active par chambre

    Exemples Concrets link

    🟢 EXEMPLE : Réservation du 15/01 au 20/01 pour une chambre disponible
   → Résultat : Réservation créée avec succès

🟢 EXEMPLE : Réservation du 20/01 au 15/01 (dates inversées)
   → Résultat : Erreur "Date de départ doit être après la date d'arrivée"

🟢 EXEMPLE : Réservation du 15/01 au 20/02 (plus de 30 jours)
   → Résultat : Erreur "Réservation ne peut pas dépasser 30 jours"

🟢 EXEMPLE : Tentative de créer une réservation pour une chambre déjà occupée
   → Résultat : Erreur "Chambre non disponible"

🟢 EXEMPLE : Tentative de modifier une réservation annulée
   → Résultat : Erreur "Réservation annulée ne peut plus être modifiée"

    Questions Émergentes link

    🔴 QUESTION : Que se passe-t-il si deux clients réservent la même chambre simultanément ?
🔴 QUESTION : Y a-t-il une période de grâce pour annuler une réservation ?
🔴 QUESTION : Peut-on déplacer une réservation vers une autre chambre ?
🔴 QUESTION : Que se passe-t-il si le client ne se présente pas ?

    Scénarios Complets link

    🔵 SCÉNARIO : Réservation standard
   1. Client consulte les disponibilités
   2. Client sélectionne une chambre libre
   3. Client saisit les dates (15/01 au 20/01)
   4. Système valide les dates
   5. Système vérifie la disponibilité
   6. Système crée la réservation
   7. Client reçoit une confirmation
   → Résultat : Réservation créée avec succès

🔵 SCÉNARIO : Annulation de réservation
   1. Client consulte ses réservations
   2. Client sélectionne une réservation active
   3. Client clique sur "Annuler"
   4. Système vérifie que la réservation peut être annulée
   5. Système marque la réservation comme annulée
   6. Système libère la chambre
   7. Client reçoit une confirmation d'annulation
   → Résultat : Réservation annulée avec succès

    Transformation en Tests d’Acceptation link

    Principe link

    Chaque exemple de l’Example Mapping doit devenir un test d’acceptation. Cela garantit que :

    • Les règles métier sont implémentées correctement
    • Les cas limites sont couverts
    • L’évolution ne casse pas les règles existantes

    Exemple de Transformation link

    Exemple Mapping → Test d’Acceptation link

    // 🟢 EXEMPLE : Paiement de 100€ avec compte ayant une limite de 500€
//   → Résultat : Paiement traité avec succès

/** @test */
public function it_processes_a_valid_payment(): void
{
    // Given
    $payment = Payment::create(
        PaymentId::generate(),
        Money::euros(100)
    );
    
    $account = Account::withLimit(Money::euros(500));
    
    // When
    $payment->process($account);
    
    // Then
    $this->assertEquals(PaymentStatus::PROCESSED, $payment->getStatus());
    $this->assertNotNull($payment->getProcessedAt());
}

    Exemple Mapping → Test d’Erreur link

    // 🟢 EXEMPLE : Paiement de -50€
//   → Résultat : Erreur "Montant invalide"

/** @test */
public function it_rejects_negative_amount(): void
{
    // Given & When & Then
    $this->expectException(InvalidAmountException::class);
    $this->expectExceptionMessage('Montant invalide');
    
    Payment::create(
        PaymentId::generate(),
        Money::euros(-50)
    );
}

    {

    }
    
    /** @test */

    {
        // Tests des cas limites
    }
}

    Bonnes Pratiques link

    ✅ À Faire link

    1. Rester Focus sur le Métier link

    • Concentrez-vous sur les règles business
    • Évitez les détails techniques
    • Gardez les exemples simples

    2. Limiter le Temps link

    • 30-45 minutes maximum par fonctionnalité
    • Arrêtez-vous quand vous avez assez d’exemples
    • Évitez l’over-engineering

    3. Valider avec l’Expert Métier link

    • Chaque règle doit être validée
    • Chaque exemple doit être réaliste
    • Les questions doivent être résolues

    4. Documenter les Décisions link

    • Notez les réponses aux questions
    • Documentez les cas limites
    • Gardez une trace des scénarios

    ❌ À Éviter link

    1. Trop de Détails Techniques link

    ❌ RÈGLE : Le système doit utiliser une base de données PostgreSQL
❌ RÈGLE : L'API doit retourner un JSON
❌ RÈGLE : Le code doit être en PHP 8.1

    2. Exemples Trop Complexes link

    ❌ EXEMPLE : Paiement de 100€ avec compte ayant une limite de 500€, 
   créé le 15/01/2024 à 14:30, avec un taux de change de 1.05, 
   en utilisant une carte Visa, avec un code de sécurité 123, 
   depuis un navigateur Chrome sur Windows 11

    3. Questions Sans Réponse link

    ❌ QUESTION : Que se passe-t-il si l'univers s'effondre ?
❌ QUESTION : Comment gérer les cas impossibles ?
❌ QUESTION : Que faire si tout échoue ?

    Intégration avec l’Event Storming link

    Workflow Complet link

    1. Event Storming (2-4 heures) link

    • Découvrir les événements métier
    • Identifier les processus
    • Définir les bounded contexts

    2. Example Mapping (30-45 min par fonctionnalité) link

    • Détailer les règles complexes
    • Créer des exemples concrets
    • Identifier les cas limites

    3. Implémentation (1-2 semaines) link

    • Transformer les exemples en tests
    • Implémenter les règles métier
    • Valider avec les tests

    Exemple de Workflow link

    Event Storming → "PaymentRequested" identifié
     ↓
Example Mapping → Règles de validation du paiement
     ↓
Tests d'Acceptation → Validation automatique
     ↓
Implémentation → Code métier robuste
