    Le Pattern Repository link

    Principe Fondamental link

    Le pattern Repository encapsule la logique d’accès aux données et fournit une interface orientée objet pour accéder aux données. Il sépare la logique métier de la logique d’accès aux données.

    Interface Repository link

    interface Repository
{
    public function save(AggregateRoot $aggregate): void;
    public function find(Id $id): ?AggregateRoot;
    public function remove(AggregateRoot $aggregate): void;
}}

interface CommandRepository extends Repository
{
    public function save(AggregateRoot $aggregate): void;
    public function find(Id $id): ?AggregateRoot;
    public function remove(AggregateRoot $aggregate): void;
}}

interface QueryRepository
{
    public function find(Id $id): ?QueryModel;
    public function findAll(Criteria $criteria): Collection;
    public function count(Criteria $criteria): int;
}

    Implémentation de Base link

    abstract class AbstractRepository implements CommandRepository
{
    public function __construct(
        protected EntityManagerInterface $entityManager,
        protected EventBus $eventBus
    ) {}

    public function save(AggregateRoot $aggregate): void
    {
        $this->entityManager->persist($aggregate);
        
        // Publier les événements
        foreach ($aggregate->getEvents() as $event) {
            $this->eventBus->publish($event);
        }
        
        $this->entityManager->flush();
    }

    public function find(Id $id): ?AggregateRoot
    {
        return $this->entityManager->find($this->getEntityClass(), $id);
    }

    public function remove(AggregateRoot $aggregate): void
    {
        $this->entityManager->remove($aggregate);
        $this->entityManager->flush();
    }

    abstract protected function getEntityClass(): string;
}

    Types de Repositories link

    1. Repository SQL (Doctrine ORM) link

    class PaymentRepository extends AbstractRepository
{
    protected function getEntityClass(): string
    {
        return Payment::class;
    }

    public function findByStatus(PaymentStatus $status): Collection
    {
        return $this->entityManager
            ->createQueryBuilder()
            ->select('p')
            ->from(Payment::class, 'p')
            ->where('p.status = :status')
            ->setParameter('status', $status)
            ->getQuery()
            ->getResult();
    }

    public function findPendingPayments(): Collection
    {
        return $this->findByStatus(PaymentStatus::PENDING);
    }
}

    2. Repository API (Services Externes) link

    class UserRepository implements CommandRepository
{
    public function __construct(
        private KeycloakClient $keycloakClient,
        private EventBus $eventBus
    ) {}

    public function save(AggregateRoot $user): void
    {
        if ($user->isNew()) {
            $this->keycloakClient->createUser($user->toArray());
        } else {
            $this->keycloakClient->updateUser($user->getId(), $user->toArray());
        }
        
        // Publier les événements
        foreach ($user->getEvents() as $event) {
            $this->eventBus->publish($event);
        }
    }

    public function find(Id $id): ?AggregateRoot
    {
        $userData = $this->keycloakClient->getUser($id->toString());
        
        if (!$userData) {
            return null;
        }
        
        return User::fromArray($userData);
    }
}

    3. Repository ElasticSearch link

    class PaymentQueryRepository implements QueryRepository
{
    public function __construct(
        private ElasticsearchClient $client
    ) {}

    public function find(Id $id): ?PaymentQuery
    {
        $response = $this->client->get([
            'index' => 'payments',
            'id' => $id->toString()
        ]);

        if (!$response['found']) {
            return null;
        }

        return PaymentQuery::fromArray($response['_source']);
    }

    public function findAll(Criteria $criteria): Collection
    {
        $query = [
            'index' => 'payments',
            'body' => [
                'query' => $this->buildQuery($criteria),
                'sort' => $this->buildSort($criteria),
                'size' => $criteria->getLimit(),
                'from' => $criteria->getOffset()
            ]
        ];

        $response = $this->client->search($query);
        
        return $this->hydrateCollection($response['hits']['hits']);
    }
}

    4. Repository In-Memory (Tests) link

    class InMemoryPaymentRepository implements CommandRepository
{
    private array $payments = [];

    public function save(AggregateRoot $payment): void
    {
        $this->payments[$payment->getId()->toString()] = $payment;
    }

    public function find(Id $id): ?AggregateRoot
    {
        return $this->payments[$id->toString()] ?? null;
    }

    public function remove(AggregateRoot $payment): void
    {
        unset($this->payments[$payment->getId()->toString()]);
    }

    public function clear(): void
    {
        $this->payments = [];
    }
}

    Gestion des Événements link

    Pattern Event Sourcing link

    class EventSourcedRepository implements CommandRepository
{
    public function __construct(
        private EventStore $eventStore,
        private EventBus $eventBus
    ) {}

    public function save(AggregateRoot $aggregate): void
    {
        $events = $aggregate->getEvents();
        
        if (empty($events)) {
            return;
        }
        
        $this->eventStore->append(
            $aggregate->getId(),
            $events,
            $aggregate->getVersion()
        );
        
        // Publier les événements
        foreach ($events as $event) {
            $this->eventBus->publish($event);
        }
    }

    public function find(Id $id): ?AggregateRoot
    {
        $events = $this->eventStore->getEvents($id);
        
        if (empty($events)) {
            return null;
        }
        
        return $this->reconstituteFromEvents($events);
    }

    private function reconstituteFromEvents(array $events): AggregateRoot
    {
        $aggregate = $this->createEmptyAggregate();
        
        foreach ($events as $event) {
            $aggregate->apply($event);
        }
        
        return $aggregate;
    }
}

    Projections de Lecture link

    class PaymentProjection
{
    public function __construct(
        private PaymentQueryRepository $repository
    ) {}

    public function handle(PaymentProcessed $event): void
    {
        $query = new PaymentQuery(
            $event->paymentId,
            $event->amount,
            PaymentStatus::PROCESSED,
            $event->processedAt,
            $event->processedAt
        );
        
        $this->repository->save($query);
    }

    public function handle(PaymentFailed $event): void
    {
        $query = $this->repository->find($event->paymentId);
        
        if ($query) {
            $query = $query->withStatus(PaymentStatus::FAILED);
            $this->repository->save($query);
        }
    }
}

    🔒 Transaction Management link

    Gestion des Transactions link

    class PaymentService
{
    public function __construct(
        private PaymentRepository $paymentRepository,
        private TransactionManager $transactionManager
    ) {}

    public function processPayment(PaymentId $id, Money $amount): void
    {
        $this->transactionManager->execute(function() use ($id, $amount) {
            $payment = $this->paymentRepository->find($id);
            
            if (!$payment) {
                throw new PaymentNotFoundException();
            }
            
            $payment->process($amount);
            $this->paymentRepository->save($payment);
        });
    }
}

    Transactions Distribuées (Saga Pattern) link

    class OrderSaga
{
    public function __construct(
        private EventBus $eventBus,
        private TransactionManager $transactionManager
    ) {}

    public function handle(OrderCreated $event): void
    {
        $this->transactionManager->execute(function() use ($event) {
            // Créer le paiement
            $payment = Payment::create($event->orderId, $event->total);
            $this->paymentRepository->save($payment);
            
            // Publier l'événement
            $this->eventBus->publish(new PaymentCreated($payment->getId()));
        });
    }
}

    class PaymentRepositoryTest extends TestCase
{
    private InMemoryPaymentRepository $repository;
    private EventBus $eventBus;

    protected function setUp(): void
    {
        $this->repository = new InMemoryPaymentRepository();
        $this->eventBus = new InMemoryEventBus();
    }

    #[Test]
    public function it_can_save_payment(): void
    {
        $payment = Payment::create(
            PaymentId::generate(),
            new Money(100.0, Currency::EUR)
        );

        $this->repository->save($payment);

        $this->assertNotNull($this->repository->find($payment->getId()));
    }

    #[Test]
    public function it_publishes_events_when_saving(): void
    {
        $payment = Payment::create(
            PaymentId::generate(),
            new Money(100.0, Currency::EUR)
        );

        $this->repository->save($payment);

        $this->assertCount(1, $this->eventBus->getPublishedEvents());
    }
}

    Tests d’Intégration link

    class PaymentRepositoryIntegrationTest extends TestCase
{
    private PaymentRepository $repository;
    private EntityManagerInterface $entityManager;

    protected function setUp(): void
    {
        $this->entityManager = $this->getEntityManager();
        $this->repository = new PaymentRepository($this->entityManager, new InMemoryEventBus());
    }

    #[Test]
    public function it_can_persist_payment_to_database(): void
    {
        $payment = Payment::create(
            PaymentId::generate(),
            new Money(100.0, Currency::EUR)
        );

        $this->repository->save($payment);
        $this->entityManager->clear();

        $retrieved = $this->repository->find($payment->getId());
        
        $this->assertNotNull($retrieved);
        $this->assertEquals($payment->getId(), $retrieved->getId());
    }
}

    🎯 Votre Prochaine Étape link

    Maintenant que vous comprenez les patterns de repository, quel type de stockage utilisez-vous principalement ?

    Option A : Base de données SQL (PostgreSQL, MySQL, etc.) link

    Vous utilisez une base de données relationnelle

    Caractéristiques :

    • Données relationnelles
    • ACID transactions
    • Requêtes SQL complexes
    • Performance prévisible

    Avantages : Mature, fiable, SQL puissant Inconvénients : Scaling horizontal limité

    → Aller au Chapitre 12 (Stockage SQL - Approche Classique)

    Option B : APIs Externes (Keycloak, services tiers, microservices) link

    Vous utilisez des services externes

    Caractéristiques :

    • Données distribuées
    • Intégrations multiples
    • Services spécialisés
    • Latence réseau

    Avantages : Services spécialisés, pas de duplication Inconvénients : Dépendance externe, latence

    → Aller au Chapitre 16 (Stockage API - Approche Classique)

    Option C : ElasticSearch link

    Vous utilisez ElasticSearch pour la recherche

    Caractéristiques :

    • Recherche complexe
    • Analytics et reporting
    • Grandes volumes de données
    • Requêtes non-SQL

    Avantages : Recherche puissante, analytics Inconvénients : Complexité, coût

    → Aller au Chapitre 20 (Stockage ElasticSearch - Approche Classique)

    Option D : MongoDB avec ODM Doctrine link

    Vous utilisez une base de données document NoSQL

    Caractéristiques :

    • Données semi-structurées
    • Flexibilité du schéma
    • Requêtes sur documents
    • Performance de lecture élevée

    Avantages : Flexibilité, performance, scaling horizontal Inconvénients : Pas de transactions ACID complètes

    → Aller au Chapitre 21 (Stockage MongoDB - Approche Classique)

    Option E : Stockage In-Memory link

    Vous utilisez le stockage en mémoire

    Caractéristiques :

    • Données légères et en lecture seule
    • Mise à jour uniquement lors des déploiements
    • Performance de lecture critique
    • Données de configuration ou de référence

    Avantages : Performance maximale, simplicité Inconvénients : Données perdues au redémarrage

    → Aller au Chapitre 22 (Stockage In-Memory - Approche Classique)

    Option F : Systèmes Multiples link

    Vous avez des transactions distribuées complexes

    Caractéristiques :

    • Plusieurs systèmes
    • Transactions complexes
    • Orchestration nécessaire
    • Tolérance aux pannes

    Avantages : Flexibilité maximale Inconvénients : Très complexe

    → Aller au Chapitre 23 (Stockage Complexe Temporal)

    💡 Conseil : Si vous n’êtes pas sûr, choisissez l’option A (SQL) car c’est le type de stockage le plus courant et le plus documenté.

    🔄 Alternative : Si vous voulez tout voir dans l’ordre, continuez avec le Chapitre 7.
